## 1. System Overview

The OLS Research System is an open-science AI research infrastructure designed for the Optimal Living Systems nonprofit. The architecture is:

- **Fine-tuned model:** Qwen 2.5 14B with QLoRA — trained on research *methodology* (how to reason about research), not domain facts
- **RAG layer:** 4 domain-specific LanceDB vector collections (psychology, sociology, environmentalism, neuroscience)
- **Embeddings:** BGE-M3 (1024-dimensional, multilingual, state-of-the-art)
- **Orchestration:** Kestra (workflow automation for collection, processing, training, evaluation)
- **Data sources:** OpenAlex (live, free API) + Semantic Scholar + PubMed OA (Phase 2)
- **Purpose:** Enable systematic, citation-grounded cross-domain research synthesis

### 4.2 GitHub CLI Setup

`gh` was not installed and `sudo apt install gh` was unavailable (no interactive sudo). **Solution:** downloaded the binary directly from GitHub releases without sudo.

```bash
cd /tmp
GH_VERSION=$(curl -s https://api.github.com/repos/cli/cli/releases/latest | grep '"tag_name"' | sed 's/.*"v\([^"]*\)".*/\1/')
curl -sL "https://github.com/cli/cli/releases/download/v${GH_VERSION}/gh_${GH_VERSION}_linux_amd64.tar.gz" -o gh.tar.gz
tar -xzf gh.tar.gz
cp gh_${GH_VERSION}_linux_amd64/bin/gh ~/.local/bin/gh
chmod +x ~/.local/bin/gh
```

**gh CLI location:** `~/.local/bin/gh` (version 2.87.3)

**Important:** `~/.local/bin` must be in your PATH. Add this to `~/.bashrc` if not already there:
```bash
export PATH="/home/joel/.local/bin:$PATH"
```

When calling `gh` from scripts, always prefix with:
```bash
export PATH="/home/joel/.local/bin:$PATH"
```
Or use the full path: `/home/joel/.local/bin/gh`

**Docker Desktop specifics:**
- Docker socket: `/home/joel/.docker/desktop/docker.sock` (NOT `/var/run/docker.sock`)
- File sharing: Docker Desktop only shares `/home` by default. Paths outside `/home` (like `/tmp`, `/data`) cannot be mounted into containers without adding them to Docker Desktop → Settings → Resources → File Sharing.
- To use Docker commands: `DOCKER_HOST="unix:///home/joel/.docker/desktop/docker.sock" docker ...`

**Issues encountered during setup:**

1. **Docker socket mount denied:** Tried mounting `/home/joel/.docker/desktop/docker.sock:/var/run/docker.sock` so Kestra could run Docker tasks. Docker Desktop blocked this path. **Resolution:** Removed the Docker socket mount for now. Kestra can run Python/shell tasks but not `io.kestra.plugin.docker` tasks until this is resolved.

2. **`/tmp/kestra-wd` mount denied:** Docker Desktop doesn't share `/tmp`. **Resolution:** Changed working directory mount to `~/kestra/wd`.

3. **`restart` vs `up --force-recreate`:** `docker compose restart` only restarts the container — it does NOT pick up new environment variables. Always use `docker compose up -d --force-recreate <service>` when changing env vars.

### 4.10 Kestra Authentication Resolution

**This is critical knowledge for future sessions.**

Kestra 1.x (unlike 0.x) requires authentication for all API access. The `basicAuth.enabled: false` setting in the YAML config does NOT disable auth — it means "don't use basic auth scheme" but Kestra still protects all endpoints.

**Discovery process:**
1. All `/api/v1/*` endpoints returned `HTTP 401`
2. The `/api/v1/configs` endpoint is publicly accessible and revealed: `"isBasicAuthInitialized": false`
3. This means Kestra 1.x requires a **first-run setup via the browser UI** to create the initial admin account

**Resolution:**
1. Open http://localhost:8080 in a browser
2. Kestra presents a registration screen on first run
3. Create an admin account (email + password)
4. After account creation, `isBasicAuthInitialized` becomes `true` in `/api/v1/configs`

**Admin account created:**
- **Email:** `joelgilbert7@yahoo.com`
- The password hash is stored in the `settings` table in PostgreSQL under the key `kestra.server.basic-auth`

**Outstanding issue:** API calls with these credentials still return 401. This suggests Kestra 1.x uses a **session/token-based auth flow** rather than HTTP Basic Auth for the API, even though basic auth is the configured mechanism. The UI works fine through the browser (cookie-based session).

**Workaround for API access:** Use the Kestra CLI inside the container to manage flows directly (bypasses the API):
```bash
DOCKER_HOST="unix:///home/joel/.docker/desktop/docker.sock" \
  docker exec kestra-kestra-1 java -jar /app/kestra flow updates \
  --server http://localhost:8080 \
  --user "joelgilbert7@yahoo.com:<password>" \
  /path/to/flows/
```
